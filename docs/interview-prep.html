Basic Level — Project Overview
General questions about the project, tech stack, and purpose
Q1 What is NoteVerse? Explain the project in simple words. ▼

NoteVerse is a full-stack notes web application — think of it like a personal Notion or Google Keep, but built from scratch with modern technologies.

It allows users to:

  • Create, edit, and organize notes using a rich text editor
  • Organize notes into folders and tag them
  • Share notes with others via a public link
  • Collaborate with teammates (give them read or write access)
  • Export notes as PDF, Markdown, or plain text
  • Search through all their notes instantly
  • Use it offline (PWA support)

The project is split into two parts: a React frontend (what the user sees) and a Spring Boot backend (the server that stores and manages data).

When answering this, mention it's a SaaS-style application — it has user accounts, authentication, and multi-user support.

Q5 What is MongoDB and why was it chosen over MySQL/PostgreSQL? ▼

MongoDB is a NoSQL database that stores data as JSON-like documents instead of rows and columns.

Why MongoDB was the right choice for NoteVerse:

  • Flexible schema — A note can have different fields: some locked, some with collaborators, some with tags. MongoDB handles this naturally without migrations.
  • Nested data — Collaborators are stored inside the Note document itself (embedded array), which is natural in MongoDB but would require a JOIN table in SQL.
  • Rich content — Note content is stored as JSON (TipTap format), which MongoDB handles natively.
  • Scalability — MongoDB Atlas scales automatically in the cloud.

Example — a Note document in MongoDB:

{
  "_id": "abc123",
  "title": "Meeting Notes",
  "content": "...",
  "tags": ["work", "important"],
  "collaborators": [
    { "userId": "u1", "email": "bob@mail.com", "permission": "READ" }
  ],
  "isPinned": true,
  "isDeleted": false
}

Q6 How is state managed in NoteVerse frontend? ▼

NoteVerse uses two layers of state management:

1. React Context (Global UI State)

  • AuthContext — stores the logged-in user, access token, login/logout functions
  • ThemeContext — stores dark/light mode preference, saves to localStorage
  • NoteContext — stores notes, folders, tags and their CRUD operations

2. TanStack React Query (Server State)

For API data fetching, caching, and synchronization. Instead of manually calling APIs and storing results in useState, React Query handles it automatically with smart caching (5 minute stale time).

// Example: NoteContext
const AuthContext = createContext();

export function AuthProvider({ children }) {
  const [user, setUser] = useState(null);
  const [token, setToken] = useState(localStorage.getItem('accessToken'));

  const login = (userData, accessToken, refreshToken) => {
    setUser(userData);
    setToken(accessToken);
    localStorage.setItem('accessToken', accessToken);
  };

  return <AuthContext.Provider value={{ user, token, login, logout }}>
    {children}
  </AuthContext.Provider>;
}
Q7 What is the difference between PrivateRoute and PublicRoute? ▼

These are custom route wrappers that control access based on authentication:

PrivateRoute

Only logged-in users can access it. If not logged in → redirect to /login.

// Example usage
<Route path="/dashboard" element={
  <PrivateRoute><DashboardPage /></PrivateRoute>
} />

PublicRoute

Only logged-out users can access it. If already logged in → redirect to /dashboard. (Prevents a logged-in user from seeing the login page again.)

// Example usage
<Route path="/login" element={
  <PublicRoute><LoginPage /></PublicRoute>
} />
This is a very common React interview question. The key point is that these are just wrapper components that check auth state and either render children or redirect.
Q8 How does the Axios interceptor work in NoteVerse? ▼

An interceptor is middleware that runs before every request is sent or before every response is received. NoteVerse uses two interceptors:

Request Interceptor — auto-add JWT token

Before every API call, it reads the access token from localStorage and adds it to the Authorization header. You don't need to manually add the token in every API call.

axiosInstance.interceptors.request.use((config) => {
  const token = localStorage.getItem('accessToken');
  if (token) config.headers.Authorization = `Bearer ${token}`;
  return config;
});

Response Interceptor — auto-refresh expired tokens

If the server returns 401 (token expired), it automatically:

  1. Calls /api/auth/refresh with the refresh token
  2. Gets a new access token
  3. Retries the original failed request
  4. If refresh also fails → logs out and redirects to /login
This creates a seamless experience — the user never sees "please log in again" unless the refresh token also expires (7 days).
Q9 What is TipTap and why was it used for the note editor? ▼

TipTap is a headless rich text editor built on ProseMirror. "Headless" means it has no default styling — you style it yourself with Tailwind CSS.

Why TipTap instead of a simple textarea:

  • Extensible — You can add custom extensions like image upload, mentions, tables
  • React integration — Works perfectly as a React component
  • Collaborative ready — Can be extended with real-time collaboration (Y.js)

Q11 What is a PWA? How is it implemented in NoteVerse? ▼

PWA (Progressive Web App) — a web app that works like a native app. It can be installed on your phone, work offline, and load fast.

NoteVerse PWA features:

  • manifest.json — defines app name, icon, theme color, so it can be "installed" on home screen
  • Service Worker — a background script that caches API responses and static assets so the app loads even without internet
  • IndexedDB (via idb) — stores notes locally in the browser database when offline
  • Sync Queue — when you edit offline, changes are queued. When internet returns, they sync to the server
A Service Worker is like a proxy between your app and the network. It intercepts requests and serves cached responses when offline.

Q15 What is a DTO and why is it used? ▼

DTO (Data Transfer Object) — a simple object used to carry data between layers. It's NOT the same as the Model/Entity.

Why use DTOs?

  • Security — Don't expose sensitive fields. The User model has `password` and `twoFactorSecret` — the UserResponse DTO excludes these.
  • Validation — DTOs have @Valid annotations to validate incoming data before processing.
  • Flexibility — API response shape can differ from database shape.
// Model — stored in MongoDB
public class User {
    private String password;      // stored in DB
    private String twoFactorSecret; // stored in DB
    private String email;
}

// Response DTO — sent to frontend
public class UserResponse {
    // No password, no twoFactorSecret!
    private String id;
    private String email;
    private String name;
}
Q16 How does GlobalExceptionHandler work in NoteVerse? ▼

GlobalExceptionHandler is a centralized place to handle all exceptions thrown anywhere in the application. Without it, every exception becomes an unformatted 500 error.

In NoteVerse, it uses @RestControllerAdvice which intercepts exceptions from all controllers:

@RestControllerAdvice
public class GlobalExceptionHandler {

  @ExceptionHandler(ResourceNotFoundException.class)
  public ResponseEntity<ApiResponse> handleNotFound(ResourceNotFoundException ex) {
    return ResponseEntity.status(404).body(ApiResponse.error(ex.getMessage()));
  }

  @ExceptionHandler(BadCredentialsException.class)
  public ResponseEntity<ApiResponse> handleBadCredentials(...) {
    return ResponseEntity.status(401).body(ApiResponse.error("Invalid email or password"));
  }

  @ExceptionHandler(Exception.class) // catch-all
  public ResponseEntity<ApiResponse> handleGeneric(Exception ex) {
    return ResponseEntity.status(500).body(ApiResponse.error("Unexpected error"));
  }
}
The order matters — more specific handlers are checked first, the generic Exception.class handler is always last.
Q17 How does soft delete work for notes? ▼

Soft delete means the data is NOT actually removed from the database. Instead, a flag is set to mark it as deleted. The user sees it in the "Trash" and can restore it.

How it works in NoteVerse:

The Note model has two fields:

private boolean isDeleted = false;
private LocalDateTime deletedAt;  // when it was deleted

When user clicks "Delete":

// NoteService.deleteNote()
note.setDeleted(true);
note.setDeletedAt(LocalDateTime.now());
noteRepository.save(note); // NOT removed from DB

When fetching notes, deleted ones are filtered out:

// NoteRepository
List<Note> findByUserIdAndIsDeletedFalse(String userId);

Permanent delete actually removes from DB:

noteRepository.delete(note); // actually removed

Notes in trash are auto-deleted after 30 days (shown as a warning in the UI).

Q18 What is JWT and how does it work in NoteVerse? ▼

JWT (JSON Web Token) is a compact, self-contained way to transmit authentication information between client and server. It does NOT require storing session data on the server.

JWT Structure (3 parts separated by dots):

header.payload.signature
eyJhbGciOiJIUzI1NiJ9.eyJzdWIiOiJ1c2VyQGV4LmNvbSJ9.xyzSignature
  • Header — algorithm used (HMAC-SHA256)
  • Payload — data: email (subject), issued time, expiry time, token type
  • Signature — Header + Payload signed with JWT_SECRET. Ensures data hasn't been tampered.

NoteVerse uses TWO tokens:

TokenExpiryPurpose
Access Token15 minutesUsed in every API request header
Refresh Token7 daysUsed to get a new access token when it expires

Q19 How does the JwtAuthFilter work step by step? ▼

The JwtAuthFilter runs on EVERY incoming HTTP request before it reaches the controller. It checks if the request has a valid JWT token.

Step-by-step flow:

  1. Read the Authorization header from the request
  2. If header is missing or doesn't start with "Bearer " → skip (let Spring Security handle it)
  3. Extract the JWT token (remove "Bearer " prefix)
  4. Extract the email from the token's payload
  5. Load the user from the database using that email
  6. Validate: token signature correct? not expired?
  7. If valid → set the user as authenticated in SecurityContext
  8. Continue to the controller
// JwtAuthFilter.java (simplified)
String authHeader = request.getHeader("Authorization");
if (authHeader == null || !authHeader.startsWith("Bearer ")) {
    filterChain.doFilter(request, response); // skip
    return;
}
String jwt = authHeader.substring(7);
String email = jwtUtils.extractUsername(jwt);
UserDetails user = userDetailsService.loadUserByUsername(email);
if (jwtUtils.validateToken(jwt, user)) {
    // Set authenticated — now controller sees logged-in user
    SecurityContextHolder.getContext().setAuthentication(authToken);
}
Q20 What is BCrypt and why is it used for passwords? ▼

BCrypt is a password hashing algorithm. Instead of storing "password123" in the database, NoteVerse stores something like $2a$10$xyz...randomhash.

Why BCrypt?

  • One-way — You can't reverse a BCrypt hash to get the original password
  • Salt included — Every hash is unique even for the same password (prevents rainbow table attacks)
  • Slow by design — Takes ~100ms to hash, making brute force very slow
  • Industry standard — Used by Spring Security by default
// Register — hash the password before saving
user.setPassword(passwordEncoder.encode(request.getPassword()));

// Login — compare without unhashing
boolean matches = passwordEncoder.matches(rawPassword, hashedPassword);
// BCrypt compares safely — never decrypts
Never store plain text passwords. If your database is leaked, BCrypt-hashed passwords are extremely hard to crack.
Q21 What is CORS and how is it configured in NoteVerse? ▼

CORS (Cross-Origin Resource Sharing) is a browser security feature. By default, a webpage at frontend.com cannot make API calls to backend.com — they're different origins.

What is "origin"?

Origin = protocol + domain + port. These are different origins:

  • https://noteverse-1.onrender.com (frontend)
  • https://noteverse-8e4i.onrender.com (backend)

How NoteVerse configures CORS:

@Bean
public CorsConfigurationSource corsConfigurationSource() {
    CorsConfiguration config = new CorsConfiguration();
    config.setAllowedOrigins(List.of("https://noteverse-1.onrender.com"));
    config.setAllowedMethods(List.of("GET","POST","PUT","DELETE","OPTIONS"));
    config.setAllowedHeaders(List.of("*"));
    config.setAllowCredentials(true); // allow cookies/auth headers
    // ...
}

The frontend URL comes from the FRONTEND_URL environment variable — no hardcoding. NoteVerse also strips trailing slashes to prevent CORS mismatch bugs.

Q23 How does password reset work using OTP? ▼

Step-by-step flow:

  1. User enters their email on the Forgot Password page
  2. Backend generates a random 6-digit OTP (using SecureRandom)
  3. OTP is stored in an in-memory map with the email as key and 10-minute expiry
  4. Email is sent with the OTP (via Spring Mail / Gmail SMTP)
  5. User enters the OTP + new password
  6. Backend validates: does OTP exist? is it expired? does it match?
  7. If valid → hash and save new password, invalidate all active sessions
// OTP storage (in-memory)
private final Map<String, OtpEntry> otpStore = new ConcurrentHashMap<>();

// Generate OTP
String otp = String.valueOf(100000 + random.nextInt(900000)); // 6-digit
otpStore.put(email, new OtpEntry(otp)); // expires in 10 min
emailService.sendPasswordResetEmail(email, name, otp);
The OTP is stored in-memory (ConcurrentHashMap), not in the database. This means if the server restarts, all OTPs are lost. In production, Redis would be better.

Q25 How does Spring Data MongoDB repository work? What is the query derivation? ▼

Spring Data MongoDB automatically generates database queries from method names — you don't write any MongoDB query code!

public interface NoteRepository extends MongoRepository<Note, String> {

    // Finds notes where userId = ? AND isDeleted = false
    List<Note> findByUserIdAndIsDeletedFalse(String userId);

    // Finds notes where 'collaborators' array contains {userId: ?}
    List<Note> findByCollaboratorsUserIdAndIsDeletedFalse(String userId);

    // Full-text search in title or content
    List<Note> findByUserIdAndTitleContainingIgnoreCase(String userId, String keyword);

    // Find deleted notes for trash page
    List<Note> findByUserIdAndIsDeletedTrue(String userId);
}

Spring reads the method name and generates MongoDB queries automatically:

  • findBy → WHERE clause
  • And → AND condition
  • IsDeletedFalse → { isDeleted: false }
  • ContainingIgnoreCase → regex search, case-insensitive
Q26 How is collaboration implemented? How are collaborators stored? ▼

Collaborators are stored as an embedded array inside the Note document — no separate collection needed.

// Note.java model
private List<Collaborator> collaborators = new ArrayList<>();

// Collaborator embedded object
public class Collaborator {
    private String userId;
    private String email;
    private String permission; // "READ" or "WRITE"
    private LocalDateTime addedAt;
}

Access control logic:

private Note findNoteWithAccess(String noteId, String userId, String required) {
    Note note = noteRepository.findById(noteId)...;

    // Owner always has full access
    if (note.getUserId().equals(userId)) return note;

    // Check collaborator list
    boolean allowed = note.getCollaborators().stream().anyMatch(c ->
        c.getUserId().equals(userId) &&
        ("READ".equals(required) || "WRITE".equals(c.getPermission()))
    );
    if (allowed) return note;

    throw new UnauthorizedException("No permission");
}
Embedding collaborators in the Note document is an intentional MongoDB design decision. It avoids JOINs and makes access checks fast (single document read).

Q28 Why is the app stateless? What does stateless mean? ▼

Stateless means the server does NOT remember anything between requests. Every request must carry all the information needed to process it — including the JWT token.

Stateful (traditional sessions) vs Stateless (JWT):

.sessionManagement(session ->
    session.sessionCreationPolicy(SessionCreationPolicy.STATELESS)

Stateless architecture is key for scaling to multiple server instances. Any instance can handle any request since there's no shared session state.
Q29 How does the share link feature work end-to-end? ▼

Creating a share link:

  1. Owner clicks Share → sets permission (VIEW/EDIT) and optional expiry date
  2. Backend generates a random UUID token (32 hex chars)
  3. A SharedLink document is saved in MongoDB: { noteId, token, permission, expiresAt, isActive: true }
  4. A public URL is returned: https://frontend.com/share/{token}

Accessing a shared note:

  1. Anyone opens the share URL (no login required)
  2. Frontend calls GET /api/share/public/{token} (public endpoint, no JWT needed)
  3. Backend finds the SharedLink by token
  4. Validates: is it active? is it expired?
  5. Returns the note content if valid
  6. Increments viewCount on the note
// SharedLink document
{
  "noteId": "abc123",
  "token": "a1b2c3d4...", // random, hard to guess
  "permission": "VIEW",
  "expiresAt": "2024-12-31T23:59:59",
  "isActive": true
}
Q30 How does the refresh token flow prevent the user from being logged out frequently? ▼

Access tokens expire in 15 minutes for security. Without refresh tokens, users would have to log in every 15 minutes. Refresh tokens solve this silently.

Silent token refresh flow:

User makes API call with expired access token
↓
Server returns 401 Unauthorized
↓
Axios interceptor catches 401
↓
Automatically calls POST /api/auth/refresh with refresh token
↓
Server validates refresh token, issues new access token
↓
Original request is retried with new access token
↓
User sees nothing — seamless experience ✓

The refresh token itself lasts 7 days. If you don't open the app for 7 days, you'll need to log in again.

Q34 Explain the Dockerfile used in NoteVerse. What is a multi-stage build? ▼

NoteVerse uses a multi-stage Docker build — two separate stages in one Dockerfile to keep the final image small.

# Stage 1: BUILD — has Maven + JDK (heavy, ~600MB)
FROM maven:3.9.6-eclipse-temurin-17 AS build
WORKDIR /app
COPY pom.xml .
RUN mvn dependency:go-offline -q  # cache dependencies
COPY src ./src
RUN mvn clean package -DskipTests -q  # compile & build JAR

# Stage 2: RUN — only JRE (lightweight, ~100MB)
FROM eclipse-temurin:17-jre-alpine
WORKDIR /app
COPY --from=build /app/target/noteverse-backend-*.jar app.jar
EXPOSE 8080
ENTRYPOINT ["java", "-jar", "app.jar"]

Why multi-stage?

Single StageMulti-Stage
Image size~700MB (Maven + JDK)~130MB (JRE only)
SecurityBuild tools exposedOnly runtime, minimal surface
Deploy speedSlow (large image)Fast (small image)
Q35 How does the GitHub Actions CI/CD pipeline work? ▼

NoteVerse uses GitHub Actions for Continuous Integration (CI) and Continuous Deployment (CD).

Pipeline flow on every push to main:

1. Push code to GitHub main branch
↓ triggers automatically
2. Job 1: Build Backend (Maven -DskipTests)
Job 2: Build Frontend (npm ci + npm run build) — runs in parallel
↓ both must pass
3. Job 3: Trigger Render backend deploy via webhook
Job 4: Trigger Render frontend deploy via webhook — parallel

Secrets management:

Sensitive values are stored in GitHub Repository Secrets (Settings → Secrets), not in code:

env:
  MONGODB_URI: ${{ secrets.MONGODB_URI }}
  JWT_SECRET: ${{ secrets.JWT_SECRET }}
  RENDER_BACKEND_DEPLOY_HOOK: ${{ secrets.RENDER_BACKEND_DEPLOY_HOOK }}
Never hardcode secrets in code. GitHub Secrets are encrypted and only injected at runtime in GitHub Actions.

Q37 Why does the frontend have a _redirects file? What problem does it solve? ▼

Problem: React Router handles routing on the client side (in the browser). When you navigate to /dashboard within the app, React Router handles it fine. But if you directly visit https://noteverse.onrender.com/dashboard in your browser or refresh the page, Render's CDN looks for a physical file called "dashboard" — which doesn't exist → 404 error.

Solution — _redirects file:

/*  /index.html  200

This tells Render: "For ANY URL, serve index.html and return 200 OK." React then loads, reads the URL, and React Router handles the routing in the browser.

React loads
→
React Router reads /dashboard
→
Shows Dashboard page ✓

"The most challenging part was setting up the production deployment on Render. Q41 Quick fire: What does each annotation do? (@SpringBootApplication, @Service, @Repository, @Component, @Bean) ▼
AnnotationWhat it does
@SpringBootApplicationMain entry point. Combines @Configuration + @EnableAutoConfiguration + @ComponentScan. Tells Spring to auto-configure everything and scan for components.
@ServiceMarks a class as a service (business logic layer). Spring manages it as a singleton bean. Same as @Component but more descriptive.
@RepositoryMarks a class as a data access layer. Enables exception translation (converts DB exceptions to Spring exceptions).
@ComponentGeneric Spring-managed bean. Parent of @Service, @Repository, @Controller.
@BeanUsed in @Configuration classes to declare a bean manually. Used when you can't annotate the class itself (third-party classes).
@AutowiredTells Spring to inject a dependency automatically. With @RequiredArgsConstructor, it's implicit for final fields.
@ValueInjects a value from application.yml: @Value("${app.jwt.secret}")
@RestControllerAdviceGlobal exception handler — catches exceptions from all controllers.

Q43 What is the ApiResponse wrapper class and why is it used? ▼

ApiResponse is a generic wrapper that ensures ALL API responses have the same consistent structure — whether success or error.

public class ApiResponse<T> {
    private boolean success;
    private String message;
    private T data;   // generic — works for any type
}

Why consistent responses matter:

  • Frontend code is simpler — always check response.data.success
  • Easy error handling — always check response.data.message
  • Professional API design — follows REST best practices
// Success response
{ "success": true, "message": "Note created", "data": { "id": "123", ... } }

// Error response
{ "success": false, "message": "Note not found", "data": null }
Q44 How does export to PDF work in NoteVerse? ▼

NoteVerse uses two approaches for export:

Client-side (Frontend) — jsPDF + html2canvas

  1. Take a screenshot of the rendered note HTML using html2canvas
  2. Add that image to a PDF using jsPDF
  3. Trigger browser download directly — no server involved

Server-side (Backend) — /api/export/{id}/pdf

  1. Frontend calls the export API endpoint with authentication
  2. Backend retrieves the note from MongoDB
  3. Converts TipTap JSON content to formatted text
  4. Returns the content as a downloadable file with proper Content-Type headers
// Markdown export example
@GetMapping("/{noteId}/markdown")
public ResponseEntity<byte[]> exportMarkdown(@PathVariable String noteId, ...) {
    String content = "# " + note.getTitle() + "\n\n" + note.getContent();
    byte[] bytes = content.getBytes(StandardCharsets.UTF_8);
    return ResponseEntity.ok()
        .header("Content-Disposition", "attachment; filename=note.md")
        .contentType(MediaType.TEXT_PLAIN)
        .body(bytes);
}
